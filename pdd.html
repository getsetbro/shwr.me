Prototyping - A

Prototypes are more valuable to then wireframes or designs - many requirements change once the client can test and move what they have requested. Most often the design is signed off on before anyone has thought about how it work in different widths.

A prototype will provide proof of concepts and be a source of documentation through out the project. Such as "Does this carousel work in any width?" The place to test that is the prototype.

Prototyping - B

Once delivery begins should time be spent on maintaining the prototype? The answer is "yes" if time spent on the prototype equals code used in the actual app. Otherwise the effort would be duplicated.

Example: PatternLab is a prototype + style guide in one. Shows the single components or groups and shows the code used to build them. Components can be demo'd as they are completed. PatternLab in particular helps to show if your components are ready for re-use. It shows the components alone as well as next to other components.

Prototyping - C

Proper setup of build tools means that the exact same code that builds the prototype will be used for the production site.

This might lead to the integration team finding that the components work well in the prototype but have bugs after integration. At this point the either team would make the needed changes with both integration and the prototype in mind. Remember that a prototype with something like PatternLab is continuous testing of the UI.

CSS

As a base use the latest bootstrap. Everybody learn bootstrap. Even designers, design for bootstrap grid. From there you will be able to transfer your knowledge to other UI frameworks.

If you want different margins then customize bootstrap before you download it.

Only build something new if bootstrap does not provide it - it usually does provide it.

CSS Conventions

Then when we do need to create something new use conventions. Building without a having re-use in mind is a shot in the foot.

SUIT CSS is a methodology focused on improving the CSS authoring experience for component-based development. Modularity, Cohesion, Composition and configuration, Loose coupling, Soft encapsulation. suitcss naming-conventions

For a full demo see:
http://jsbin.com/lumeqa/2/

It is a learning curve for sure but the results are worth it. This will scale from small to large projects.

JavaScript

Declarative HTML - You can tell what is by looking at the markup. Elements affected by JS should have separate css classes that look different: "js-SiteHeader" or potentially data-attributes. That way you can safely change style classes but carefully change JS classes.

class="icon-class js-icon" data-icon="iconJS"
$(".js-icon").on(...);
$("[data-icon]").on(...);
